## Known bugs surfaced during pitch-shifter investigation (2026-05-18)

These were identified while diagnosing the "no audio manipulation" report; the pitch-shifter fix landed in the same session, these two are open and need their own commits.

- [x] **Transport stop + restart leaves audio path silent.** ✅ 2026-05-18 — Root cause: `VideoElementAudioSource.dispose()` disconnected the cached `MediaElementAudioSourceNode`, but `AudioEngine.setSource()` calls `old.output.disconnect()` *before* swapping and the new `VideoElementAudioSource` shares the same cached node (the node cannot be recreated for the same `<video>`). Sequence was: disconnect → swap → `#rewire()` reconnects → `old.dispose()` disconnects again, after the rewire, leaving the chain silent. Fix: `VideoElementAudioSource.dispose()` is now a no-op (the cached node is owned by the per-AudioContext `WeakMap` for the lifetime of the context; the engine already handles disconnect via `old.output.disconnect()` before the swap). Verified via Playwright: full stop → restart cycle now returns peak FFT to -47 dBFS (≈baseline -51), pre-fix was -120 dBFS silent.
- [x] **`__AV_SYNTH_QA__.setOperatorParam` silently accepts invalid paramIds.** ✅ 2026-05-18 — Guard added in `src/App.svelte` `installQaBridge`: `setOperatorParam` and `setSourceParam` both now check the paramId against `def.paramOrder`. Invalid ids return `false` and emit a `console.warn` listing the valid ids, instead of silently writing to a non-existent key. Verified: `setOperatorParam('brightness', 'brightness', 1)` now returns `false` with warn `[QA bridge] setOperatorParam: unknown paramId "brightness" for op "brightness". Valid ids: amount`.
- [x] **Manual test path was hearing dry `<video>` audio instead of the Web Audio graph.** ✅ 2026-05-19 — Root cause: the upload path and transport ownership were wrong, but the first attempted fix overcorrected by muting the hidden media element itself. Per MDN `AudioContext.createMediaElementSource()`, the media playback is already rerouted into the `AudioContext` graph, so forcing `muted = true` / `volume = 0` can suppress the graph feed as well. Final fix: on upload, immediately switch to the video source, initialise the audio graph if needed, resume the shared transport, and play the clip from that same user gesture, while leaving the media element at normal volume so the rerouted graph has signal. `Start` / `Stop` still own subsequent transport state.
